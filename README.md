# Parallel Mandelbrot Engine: An Architectural Analysis

A high-performance Mandelbrot set renderer built in C++ to demonstrate and benchmark different levels of parallel computing architectures. This project analyzes the performance gains and limitations of **Thread-Level Parallelism (MIMD)** via OpenMP and **Data-Level Parallelism (SIMD)** via auto-vectorization.

## Project Overview

The Mandelbrot set is generated by iterating the complex function `z{n+1} = z(n)^2 + c`. Because the calculation for each pixel is entirely independent of its neighbors, it is an "embarrassingly parallel" problem.

This project renders a high-resolution `.ppm` (Portable PixMap) fractal image and outputs high-precision terminal benchmarks comparing three execution models:
1. **Serial Execution (SISD):** Standard single-core processing.
2. **Multi-Threaded Execution (MIMD):** Distributed processing across CPU cores using OpenMP.
3. **SIMD + Multi-Threaded:** Attempted register-level vectorization combined with multi-threading.

## Architectural Highlights

* **Dynamic Load Balancing:** The Mandelbrot set suffers from severe workload imbalance (pixels inside the "black" set require maximum iterations, while outer pixels escape quickly). This engine uses OpenMP's `schedule(dynamic)` clause to prevent idle cores, ensuring maximum CPU utilization.
* **Cache-Friendly Data Structures:** The image is stored as a contiguous 1D `std::vector` to maximize spatial locality and minimize L1/L2 cache misses during sequential memory writes.
* **Instruction-Level Optimizations:** Built with `-O3` and `-ffast-math` compiler directives to aggressively optimize the inner floating-point loop.

## Performance Benchmarks

*Hardware Context: 2-Core Virtual Machine (Ubuntu/Arch)*
*Resolution: 1000x750 | Max Iterations: 1000*

| Execution Mode | Paradigm | Execution Time | Speedup vs Serial |
| :--- | :--- | :--- | :--- |
| **Serial** | SISD | ~2.70s | 1.00x |
| **Parallel (OpenMP)** | MIMD | ~1.26s | **2.14x (Linear Scaling)** |
| **Parallel + SIMD** | MIMD + SIMD | ~1.41s | 1.91x |

### Architectural Insight: The SIMD Bottleneck
Contrary to expectations, the SIMD implementation performs *slower* than the standard parallel implementation. This is a classic demonstration of **Branch Divergence** (Control Flow Divergence). 

Because adjacent pixels require vastly different iteration counts before escaping the set ($|z| \ge 2.0$), the vector registers (AVX/SSE) become desynced. The CPU must use vector masking to disable lanes that finish early, forcing them to sit idle while waiting for the longest-running pixel in the vector batch to finish. The overhead of this masking outweighs the benefits of the wide registers for this specific algorithm.
